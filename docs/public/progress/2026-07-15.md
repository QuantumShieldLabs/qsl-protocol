# QSL Progress — 2026-07-15

This is a catch-up entry covering two arcs merged since the 2026-07-10 entry: a
found-and-fixed handshake authentication flaw with its bounded machine-checked
follow-up, and a product-path engineering sequence. It is an engineering-evidence
summary, not a release certificate. It makes no public-readiness,
production-readiness, external-review-complete, crypto-complete,
proven-Triple-Ratchet, bug-free, or vulnerability-free claim.

## A found-and-fixed handshake authentication flaw (ENG-0038)

This is the first public Progress record of ENG-0038: an authentication flaw in
the shipped `qsc` client handshake, found by QSL's own internal adversarial
review, fixed, and then bounded-model-checked. It is published as a correction
record and a worked example of the project's review discipline — not as evidence
that the protocol is now flaw-free.

- **The finding (NA-0632, D-1256).** An internal adversarial re-analysis of the
  shipped `qsc` `QSC.HS.*` handshake found that it did not authenticate the
  responder to the initiator: an active on-path attacker could complete a
  handshake toward the initiator as if it were the intended responder, and a
  correctly verified out-of-band code did not prevent this. The finding was
  established by protocol-trace analysis (not a proof-of-concept exploit) and
  contradicted an earlier internal "no such binding flaw" conclusion
  (ENG-0001 / NA-0609B) — that contradiction is itself recorded as a calibration
  point for what internal inspection alone can miss. Record:
  `docs/governance/evidence/NA-0632_adversarial_reanalysis.md`.
- **The fix (NA-0633, D-1257, merged).** The handshake now authenticates the
  responder against the responder identity KEM key the initiator has pinned for
  that contact: the initiator encapsulates to the pinned key and the resulting
  secret is mixed into the handshake key derivation, so a responder that does
  not hold the pinned key cannot produce an acceptable reply and is rejected at
  the initiator's verification step. Merged with the full local `qsc` suite
  green. Record: `docs/governance/evidence/NA-0633_design_lock.md` and D-1257.
- **The flaw-class retirement (NA-0634, D-1258).** The surrounding
  authentication-asymmetry class was reviewed across the handshake surface and
  retired, and the follow-up verification obligation was filed rather than
  assumed. Record: `docs/governance/evidence/NA-0634_as_built.md`.
- **The bounded machine-checked model (NA-0636 / D572, D-1259).** A bounded,
  machine-checked model of the handshake's authentication slice was added and
  discharged the filed ENG-0038 obligation affirmatively within its stated
  bounds (an exhaustive search over its finite configuration space — 10,800
  responder and 10,800 initiator configurations — with zero unbound
  commitments, and a non-vacuity demonstration that the model detects the
  original flaw when the fix is rewound). The model has known unmodeled
  slices, recorded on the public improvement ledger — contact-store device
  indirection (the substantive one), cross-session replay, concurrent pending
  handshakes, composition with suite negotiation, and fingerprint
  collision-resistance (held as an assumption with a code-format tripwire).
  Records: `formal/README.md` (the model, its properties, and its limits) and
  `docs/governance/evidence/NA-0636_as_built.md`.

Reviewer-facing formal-methods records: `formal/README.md` for the handshake
authentication model above, and `docs/design/DOC-G4-002` for the separate
CI-gated ProVerif analysis of the Suite-2 DH+PQ composition (summarized in the
2026-07-10 entry).

What this arc establishes: one real flaw was found internally, fixed, its class
retired, and the fix's central property machine-checked within a bounded model.
What it does not establish: it does not establish that the protocol is
vulnerability-free or that all flaws have been found; the model is bounded, with
known unmodeled slices on the ledger, and is not an unqualified formal
verification of the handshake; the review that found the flaw was internal —
independent external review has not yet been commissioned and remains a release
gate.

## Product-path engineering (NA-0640..NA-0646)

- **Full-stack end-to-end integration (NA-0640, D-1263).** The standard test
  suite gained its first full-stack end-to-end scenario: two real `qsc` clients
  through a real `qsl-server` relay and the real `qsl-attachments` service, with
  the message round-trip byte-matched, a 6 MiB attachment byte-verified, and
  both open and bearer-token auth modes exercised plus a wrong-bearer negative
  case. This is a development-harness result under controlled conditions, not a
  deployment or service claim. Record:
  `docs/governance/evidence/NA-0640_as_built.md`.
- **Durable relay queue (NA-0642, D-1265).** `qsl-server`'s store-and-forward
  queue now persists in embedded SQLite, with an opt-in acknowledged-pull
  contract (delete only on acknowledgment), an operator-tunable retention
  window replacing the previous idle-discard, byte-identical legacy pull
  behavior, and kill-based crash-durability checks green. This is repository
  evidence about the server codebase, not a claim that any deployed relay is
  running it; the production-hardening bundle (ENG-0039) remains open. Records:
  `tests/NA-0642_qsl_server_durability_testplan.md` and
  `docs/governance/evidence/NA-0642_as_built.md`.
- **Acknowledged-pull client (NA-0644, D-1267).** `qsc` gained an opt-in
  `receive --ack-mode lease` mode that persists each message durably before
  acknowledging, with durable per-mailbox message-id deduplication so a
  redelivered message is skipped and acknowledged rather than reprocessed. The
  claim is deliberately bounded: this closes the client's pull-to-persist crash
  window except one bounded, filed seam (ENG-0042, on the public ledger), and
  lease mode is not the default. Record:
  `docs/governance/evidence/NA-0644_as_built.md`.
- **TUI retirement (NA-0645, D-1268).** The prototype terminal UI was removed
  (about 18.9k lines) after its two load-bearing capabilities were re-homed to
  the CLI and proven green first. Three security features that existed only in
  the TUI (vault attempt-limit, idle autolock, account destroy) were filed as
  ENG-0044: a future graphical client must restore them or record explicit
  per-feature drop decisions. Record:
  `docs/governance/evidence/NA-0645_as_built.md`.
- **Core extraction (NA-0646, D-1269).** The `qsc` messaging core is now a
  linkable library with no process-exit sites in the library (exit semantics
  live in a single binary-side adapter), delivered as two sequential PRs — pure
  code motion, then error-plumbing conversion — with a byte-identity prover
  showing the command-line behavior byte-identical across the change (14/14
  fixed cases, including a demonstration that the prover can fail). This is
  engineering architecture in support of a possible future in-process graphical
  client; it is not a product, SDK, or release claim. Record:
  `docs/governance/evidence/NA-0646_as_built.md`.

## Publication-time accuracy note

While preparing this publication, the public local demo script
(`scripts/demo/qsc_demo_local.sh`) was smoke-run at current `main` and found not
to deliver messages: it pre-dates the client's explicit vault-unlock requirement
and reports completion even though every send/receive fails. The failure is
recorded (`docs/governance/evidence/NA-0647_as_built.md`, and WCM-110 in the
website claim matrix) and flagged for a follow-up fix lane; it is not fixed in
this docs-only publication.

## What this does and does not establish

These are merged, in-tree engineering results: a fixed handshake flaw with a
bounded machine-checked follow-up, a byte-verified full-stack integration
scenario in a development harness, durable-queue and acknowledged-pull behavior
proven by tests, and a client-code restructuring proven behavior-preserving.
None of this changes the project's claim boundary: no public readiness, no
production readiness, no public-internet readiness, no completed external
review, no crypto-completeness, no vulnerability-free status, and no
unqualified formal-verification claim. QSL remains a research-stage,
non-production effort, and independent external review remains a release gate.

Review is invited on the evidence, limits, claim boundaries, corrections, and
residual gaps (`formal/README.md`, `docs/design/DOC-G4-002`, `DECISIONS.md`,
`TRACEABILITY.md`, `docs/ops/IMPROVEMENT_LEDGER.md`, and the release-readiness
evidence map).
